# Java 21

Use this skill to apply Java 21 features where they improve clarity, safety, or alignment with the spec.

## Default Approach

1. Read the Speckit behavior first and model the business language directly.
2. Prefer simple Java over clever Java. Use Java 21 features when they reduce accidental complexity.
3. Keep domain code framework-free.
4. Favor immutability for commands, results, value objects, and domain events.
5. Preserve existing project conventions unless they conflict with correctness or maintainability.

## Useful Java 21 Idioms

- Use records for immutable data carriers: commands, queries, responses, test fixtures, value objects without complex invariants.
- Use compact constructors in records to enforce invariants.
- Use sealed interfaces/classes for closed domain alternatives when the set is intentionally finite.
- Use switch expressions and pattern matching to make branching exhaustive and explicit.
- Use `Optional` for possibly absent return values, not for fields, parameters, or collections.
- Use `List.of`, `Set.of`, and `Map.of` for small immutable fixtures.
- Use `Stream` for transformations that stay readable; use loops when the logic is procedural or stateful.

## Records

```java
public record CreateCustomerCommand(String name, String document) {
    public CreateCustomerCommand {
        if (name == null || name.isBlank()) {
            throw new IllegalArgumentException("name must not be blank");
        }
        if (document == null || document.isBlank()) {
            throw new IllegalArgumentException("document must not be blank");
        }
    }
}
```

Use records for semantic data. Do not use them for JPA entities unless the project has a deliberate persistence strategy that supports it.

## Sealed Types

```java
public sealed interface PaymentResult permits PaymentAccepted, PaymentRejected {
}

public record PaymentAccepted(String authorizationCode) implements PaymentResult {
}

public record PaymentRejected(String reason) implements PaymentResult {
}
```

Use sealed types when the spec has a closed set of outcomes and callers should handle each one.

## Virtual Threads

- Consider virtual threads for blocking I/O workloads in adapters or application entry points.
- Do not use virtual threads as a substitute for fixing shared mutable state.
- Keep transaction, thread-local, and framework behavior in mind before changing execution models.
- Avoid introducing virtual-thread infrastructure unless the project already supports Java 21 runtime configuration.

## Error Handling

- Represent expected business rejections as domain outcomes when callers need to branch on them.
- Use exceptions for exceptional failures, invalid programmer input, or infrastructure failures.
- Translate domain/application outcomes to HTTP, messaging, or persistence responses in adapters.

## SDD Checklist

- Code names match the spec vocabulary.
- Java 21 features make behavior more explicit, not more obscure.
- Public APIs show valid states and outcomes clearly.
- Tests cover every spec branch introduced by records, sealed types, or switch expressions.
